GIT

Este documento proporciona una guía básica para usar Git en Fedora, incluyendo instalación, configuración inicial y operaciones comunes.

Instalación y Configuración

Instalar Git

sudo dnf install git

Configurar Git

git config --global user.name "nombre" git config --global user.email "correo@dominio.com"

Generar y configurar SSH Key

ssh-keygen -t rsa -b 4096 -C "correo@gmail.com" eval "$(ssh-agent -s)" ssh-add ~/.ssh/id_rsa

Añade SSH Key a GitHub

cat /root/.ssh/id_rsa.pub - Copia el contenido y añádelo a tu cuenta de GitHub.

Actualizar el sistema

sudo dnf upgrade -y

Primeros Pasos con Git

1. Crea un nuevo repositorio en GitHub.

2. Inicializa un repositorio local (si aún no existe):

git init

3. Verifica el estado del repositorio:

git status

4. Conecta el repositorio local con GitHub:

git remote add origin git@github.com:usuario/repositorio.git

5. Prepara los archivos para el commit:

git add . # Añade todos los archivos

o

git add nombre_archivo # Añade archivos específicos

6. Realiza el commit:

git commit -m "Mensaje descriptivo del commit"

7. Sube los cambios a GitHub:

git push -u origin main # La primera vez git push # Las siguientes veces

Operaciones Comunes

1. Actualizar el repositorio local

git pull origin main

2. Verificar el estado y el historial

git status # Muestra el estado actual del repositorio git log # Muestra el historial de commits

3. Cambiar la rama principal

git branch -m master main # Cambia de 'master' a 'main'

4. Forzar un push (usar con precaución)

git push -f origin main # Sobrescribe el historial remoto

5. Eliminar archivos

git rm nombre_archivo git commit -m "Eliminado nombre_archivo" git push origin main

Recordatorios Importantes

- Hacer pull antes de push para evitar conflictos.

- Usar mensajes de commit descriptivos.

- Tener cuidado con los comandos que pueden sobrescribir el historial (git push -f).

- Mantener tu repositorio local actualizado.

Errores comunes

error: remote origin already exists.

Este mensaje indica que se ha realizado una actualización forzada (forced update) en el repositorio remoto. Vamos a desglosar lo que significa:

1. branch main -> FETCH_HEAD:

- Esto indica que la información de la rama 'main' del repositorio remoto ha sido recuperada y almacenada temporalmente en FETCH_HEAD.

2.+ 400e4c6...f1ae838 main -> origin/main (forced update):

- + al principio indica una actualización forzada.

- 400e4c6...f1ae838 muestra los hashes de commit abreviados, desde el commit anterior hasta el nuevo.

- main -> origin/main indica que la rama local 'main' está siendo actualizada con los cambios de 'origin/main'.

- (forced update) confirma que esta actualización se hizo de manera forzada, posiblemente sobrescribiendo el historial existente.

Este tipo de actualización suele ocurrir cuando alguien ha hecho un git push --force en el repositorio remoto, lo que puede reescribir el historial.

Implicaciones y próximos pasos:

1. Precaución: Si tenías cambios locales que no estaban en el remoto, es posible que se hayan perdido o entrado en conflicto.

2. Revisar cambios: Es buena idea revisar los cambios que se han introducido:

git log --oneline ORIG_HEAD..

3. Actualizar tu copia local: Si estás de acuerdo con los cambios:

git pull

4. En caso de conflictos: Si tienes cambios locales que entran en conflicto, Git te lo notificará. Tendrás que resolver estos conflictos manualmente.

5. Si prefieres mantener tu versión: Puedes considerar crear una nueva rama con tus cambios antes de hacer pull:

git branch mi-version-anterior git pull

Verificar archivos

Para verificar los archivos que se encuentran en tu repositorio de GitHub desde la terminal, puedes seguir estos pasos:

1. Listar archivos en la rama actual sin descargarlos:

git ls-remote --heads origin

Esto mostrará las ramas disponibles en el repositorio remoto.

2. Para ver el contenido de una rama específica (por ejemplo, 'main'):

git ls-tree -r --name-only origin/main

Este comando lista todos los archivos en la rama 'main' del repositorio remoto.

3. Si quieres ver más detalles, incluyendo los últimos commits:

git log --name-status origin/main

4. Para ver los cambios entre tu versión local y la remota:

git fetch git diff --name-status origin/main

5. Si quieres una vista más detallada del estado del repositorio remoto:

git remote show origin

6. Para ver el contenido de un archivo específico sin descargarlo:

git show origin/main:path/to/file

Reemplaza 'path/to/file' con la ruta del archivo que quieres ver.

7. Si necesitas actualizar tu información local sobre el remoto:

git fetch origin

Esto actualiza tu información local sobre el estado del repositorio remoto sin modificar tus archivos locales.

Recuerda reemplazar 'main' con el nombre de la rama que quieras examinar si es diferente.

También, ten en cuenta que estos comandos asumen que tu repositorio remoto se llama 'origin'. Si le has dado un nombre diferente, reemplaza 'origin' con el nombre que hayas usado.

Eliminar archivos y carpetas

Para borrar archivos y carpetas de tu repositorio remoto en GitHub desde la terminal, sigue estos pasos:

1. Primero, asegúrate de que tu repositorio local esté actualizado:

git pull origin main

2. Para borrar un archivo:

git rm nombre_del_archivo

3. Para borrar una carpeta y todo su contenido:

git rm -r nombre_de_la_carpeta

4. Después de borrar los archivos o carpetas, haz un commit de estos cambios:

git commit -m "Eliminados archivos y carpetas innecesarios"

5. Finalmente, sube estos cambios al repositorio remoto:

git push origin main

Notas importantes:

- Si quieres borrar archivos o carpetas solo del repositorio remoto pero mantenerlos en tu sistema local, usa la opción --cached:

git rm --cached nombre_del_archivo git rm -r --cached nombre_de_la_carpeta

- Si los archivos que quieres borrar no están bajo control de versiones (es decir, nunca fueron añadidos a Git), simplemente elimínalos de tu sistema de archivos local y luego haz commit y push de los cambios.

- Asegúrate de que realmente quieres eliminar estos archivos o carpetas, ya que esta acción es irreversible (aunque podrías recuperarlos del historial de Git si es necesario).

- Si estás trabajando en una rama diferente a 'main', asegúrate de reemplazar 'main' con el nombre de tu rama en los comandos.
